# Codex Harness for `svgviewer`

This repository uses a long-running Codex harness. Every fresh Codex session must use the checked-in repo artifacts below rather than relying on hidden prior context.

## Source of Truth
- `AGENTS.md`: session workflow and guardrails
- `init.sh`: start or reuse the local app server
- `feature_list.json`: durable feature ledger
- `codex-progress.txt`: append-only handoff log

Do not treat `.codex/environments/environment.toml` as the durable harness source of truth. It is autogenerated convenience only.

## Start-of-Session Workflow
Run these steps in order before any implementation work:

1. Run `pwd` and confirm you are working inside this repository.
2. Read `codex-progress.txt` from top to bottom to understand the most recent handoff.
3. Read `feature_list.json` and note the highest-priority feature with `"passes": false`.
4. Read recent git history with `git log --oneline -10`.
5. Run `bash ./init.sh` from the repo root, or `bash /absolute/path/to/init.sh` from another directory.
6. Use the existing browser tooling to smoke-test the running app:
   - open the URL reported by `init.sh`
   - verify the page title is `SVG Viewer Workbench`
   - verify the default SVG preview is visible
   - verify the `React` checkbox exists
   - paste malformed SVG and confirm an error appears while the last valid preview remains visible
7. Only after the smoke test passes, choose exactly one highest-priority non-passing feature to work on.

## Choosing Work
- If `feature_list.json` contains any non-passing feature, work on exactly one of them.
- Do not work on multiple features in the same session unless the user explicitly requests it.
- If every tracked feature already passes and the user has not requested a new product change, stop and report that no harness-tracked feature remains.
- If the user explicitly requests new product work that is not yet represented in `feature_list.json`, append new feature entries first, mark them `passes: false`, then work on the single highest-priority new entry.

## Feature Ledger Rules
- `feature_list.json` must stay valid JSON.
- Existing feature objects are stable records. Do not casually rewrite or delete them.
- You may change `passes` from `false` to `true` only after verifying the feature end-to-end.
- You may change `passes` from `true` to `false` if you confirm a regression.
- When adding a new feature, include `id`, `category`, `priority`, `description`, `steps`, and `passes`.

## End-of-Session Workflow
Before ending a session:

1. Re-run the relevant verification commands for the feature you touched.
2. Update `feature_list.json` so `passes` matches the current verified state.
3. Append a new entry to `codex-progress.txt` with:
   - timestamp
   - branch
   - startup verification result
   - feature worked on
   - files touched
   - checks run
   - open issues
   - recommended next feature
4. Leave the repo in a clean, understandable state for the next session.
5. If commits are allowed and explicitly requested, use them. Otherwise, rely on git history review plus the progress log.

## Quality Bar
- Do not claim success without browser verification for user-facing behavior.
- Do not invent missing context if the repo artifacts disagree; update the artifacts instead.
- Do not mark a feature as passing based only on code inspection.
